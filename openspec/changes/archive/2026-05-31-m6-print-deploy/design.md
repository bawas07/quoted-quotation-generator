## Context

The quotation generator is feature-complete through M5: form editing, JSON import/export, catalog CRUD with autocomplete, catalog sync popup on SENT, all 5 live preview templates, history persistence, and workspace backup/restore. The app runs locally via `npm run dev`.

Two gaps remain before the app is usable in the real world:

1. **No PDF export** — the PDF button shows "coming soon" toast. `usePrint.ts` is a stub. `print.css` exists (with correct rules) but is never imported. TemplateBold and TemplateSidebar lack `@media print` blocks.
2. **No deployment** — the app only runs on `localhost`. No `_redirects` file for Cloudflare Pages SPA routing.

The design is straightforward: wire up what's already mostly built.

## Goals / Non-Goals

**Goals:**
- `window.print()` triggers on PDF button click, rendering the active template on A4
- All UI chrome (sidebar, template switcher, status bar, action buttons) hidden during print
- All 5 templates print correctly with status badge, logo, line items, and totals intact
- Colored elements (top bars, status badges, table headers) preserved via `print-color-adjust`
- App deployed to Cloudflare Pages, accessible via public URL
- localStorage persistence works on the production domain

**Non-Goals:**
- No PDF generation library (jsPDF, html2pdf, Puppeteer) — `window.print()` is sufficient for V1
- No server-side rendering or backend
- No custom print dialog or print preview UI — browser-native print dialog only
- No multi-template batch print
- No watermark or "Generated by" branding in print output
- No CI/CD pipeline — manual Cloudflare Pages connect is adequate for a solo project

## Decisions

### Decision 1: `window.print()` + CSS over PDF library

**Chosen:** Browser-native `window.print()` with `@media print` CSS.

**Alternatives considered:**
- *jsPDF / html2canvas*: Adds ~200KB dependency, requires manual canvas rendering, struggles with CSS Grid/Flexbox layouts, and produces rasterized output (blurry text). Overkill for a 5-template static print.
- *Puppeteer/headless Chrome*: Requires a server, not viable for a client-only app.

**Rationale:** `window.print()` is zero-dependency, produces vector-sharp text, respects CSS `@page` sizing, and works identically in Chrome, Firefox, and Safari. The 5 templates are already A4-proportioned (794px wide in `.invoice-paper`). Print CSS just hides the editing chrome.

### Decision 2: Scoped `@media print` per template

**Chosen:** Each template has its own `<style scoped>` block with `@media print` rules.

**Alternatives considered:**
- *Global print.css handles everything*: Can't override scoped template styles (Vue adds data attributes). Template padding and element visibility are scoped — a global stylesheet has lower specificity.
- *Unscoped styles in templates*: Loses style isolation, risking cross-template CSS leaks.

**Rationale:** Classic, Minimal, and Friendly already follow this pattern — they each have scoped `@media print` blocks adjusting padding, removing shadows, and preserving colored elements with `print-color-adjust: exact`. Bold and Sidebar need the same treatment. This keeps print rules co-located with template styles, making maintenance intuitive.

### Decision 3: `print.css` as a global stylesheet imported in `main.ts`

**Chosen:** Import `print.css` in `main.ts` alongside other global stylesheets.

**Alternatives considered:**
- *Inline in `index.html`*: Works but breaks the existing pattern (all styles imported via JS).
- *Merge into `global.css`*: Would work but print.css is a distinct concern — separating it keeps the global stylesheet lean.

**Rationale:** `print.css` is already written and covers global concerns (hiding `.sidebar`, `.preview-topbar`, `.status-bar`, `.sidebar-actions`). It's a standalone concern. Following the project's existing import pattern (`main.ts` → `global.css` → `tokens.css` + `layout.css`), adding one more import is consistent.

### Decision 4: `usePrint` composable as a thin wrapper

**Chosen:** `usePrint()` returns `{ triggerPrint }` where `triggerPrint()` calls `window.print()`.

**Alternatives considered:**
- *Direct `window.print()` in App.vue*: Works but breaks the project pattern — all side effects go through composables.
- *Complex composable with print preview state*: YAGNI. The browser handles preview natively.

**Rationale:** The composable pattern is already established (every feature has a `useX` composable). Keeping `usePrint` thin avoids over-engineering. If V2 adds custom print settings (template selection, page range), the composable is the natural place to extend.

### Decision 5: Cloudflare Pages with `_redirects` for SPA

**Chosen:** Static file deploy to Cloudflare Pages with `_redirects` file for client-side routing.

**Alternatives considered:**
- *Netlify / Vercel*: Similar free tiers, but Cloudflare Pages is already used for the sibling invoice app (consistent).
- *GitHub Pages*: No `_redirects` support without a 404.html hack. Cloudflare is cleaner.

**Rationale:** Cloudflare Pages is free, fast, and the user's existing choice. The `_redirects` file (`/* /index.html 200`) is the standard Cloudflare pattern for SPAs — zero configuration needed on the dashboard side.

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| Browser print dialogs differ (Chrome vs Firefox vs Safari) — margins, background colors, header/footer defaults | `@page { margin: 0 }` and `print-color-adjust: exact` handle most differences. Testing on Chrome + Firefox covers the primary targets. |
| Long quotations (20+ line items) may break awkwardly across pages | Existing `page-break-inside: avoid` on `tr` and `thead/tfoot` display properties handle this. Verified in Classic template; Bold and Sidebar will inherit the same rules. |
| `window.print()` blocks the UI thread (synchronous) — user can't interact with the app while the print dialog is open | This is standard browser behavior. No workaround needed — the user is in "print mode." |
| Cloudflare Pages may cache old assets after deploy | Standard: new deploy creates a new build with unique asset hashes (Vite does this by default). No cache-busting needed. |
| localStorage may not persist if the production domain differs from development | Test on the production URL immediately after deploy. Cloudflare Pages assigns a `*.pages.dev` subdomain — same origin policy applies normally. |
